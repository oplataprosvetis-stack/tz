# Часть 3. Архитектура

## 1. Каталог на миллион SKU — хранение и быстрый поиск

При миллионе товаров реляционная БД (MySQL/PostgreSQL) остаётся основным хранилищем, но критически важна правильная индексация. Составные индексы строятся по принципу «equality-колонки первыми, range-колонка последней»: `(category_id, in_stock, price)` — такой индекс покрывает самый частый сценарий фильтрации одним B-tree scan. Для полнотекстового и фасетного поиска (по множеству свойств одновременно) используется Elasticsearch — товары индексируются в него асинхронно, а поисковые запросы идут напрямую в ES, минуя MySQL. Пагинация — через `cursor-based` (keyset) или `simplePaginate`, потому что `OFFSET` + `COUNT(*)` на миллионе строк — дорогая операция. Денормализация часто используемых полей (например, `in_stock` как boolean прямо в таблице `products`, а не вычисление через JOIN со складом) снижает нагрузку на read-запросы.

## 2. Кэширование

Кэшировать имеет смысл три уровня данных. Первый — каталожные страницы и результаты фильтрации: Redis с TTL 5–15 минут, ключ формируется из параметров запроса (`products:cat=5:price=100-500:page=2`). Второй — карточки товаров: тегированный кэш Bitrix (компонентный кэш) с тегом `product_{id}`, при обновлении товара инвалидируется только его тег. Третий — справочные данные (дерево категорий, бренды): Redis или Memcached с длинным TTL и ручным сбросом при изменении.

Для инвалидации при изменении одного товара используется событийная модель: обработчик `OnAfterIBlockElementUpdate` сбрасывает кэш конкретного товара по тегу и удаляет из Redis ключи, содержащие его категорию. Массовую инвалидацию (импорт каталога) лучше делать полным сбросом кэша после завершения импорта, а не поэлементно.

## 3. Связка Bitrix ↔ Laravel ↔ Bitrix24

Данные делятся по принципу «источник правды» (source of truth). Товары: источник правды — Bitrix (инфоблоки) или ERP поставщика, откуда они импортируются. Laravel получает товары через синхронизацию и хранит свою копию для API. Заказы: источник правды — Bitrix (модуль Sale), Bitrix24 CRM получает данные о заказах асинхронно для работы менеджеров.

Синхронно делаем только то, что нужно для отклика пользователя: проверка остатков, расчёт цены. Всё остальное — через очередь: отправка заказа в Laravel, создание/обновление сделки в Bitrix24 CRM, обновление остатков после оплаты. В качестве брокера подходит Redis Queue (встроен в Laravel) или RabbitMQ для более высоких нагрузок. Связь с Bitrix24 — через REST API с OAuth, вызовы обёрнуты в retry-логику с экспоненциальной задержкой.

## 4. Импорт каталога

Импорт сотен тысяч позиций должен быть инкрементальным и неблокирующим. Файл от поставщика (CSV/XML) обрабатывается чанками по 500–1000 записей с использованием `UPSERT` (INSERT ... ON DUPLICATE KEY UPDATE), что позволяет обновлять существующие и добавлять новые товары одним запросом. На время импорта отключаются обработчики событий Bitrix (`OnAfterIBlockElementUpdate` и т.п.) и пересчёт фасетных индексов — они пересчитываются один раз по завершении.

Импорт запускается в off-peak часы (ночь) через cron или Laravel Artisan command. Для защиты сайта от просадки используются: ограничение скорости вставки (sleep между чанками), отдельное соединение к БД с пониженным приоритетом, и флаг «импорт в процессе», который переключает кэш на более длинный TTL. После завершения — полный сброс кэша каталога и переиндексация Elasticsearch (если используется).
